Jaros records every decision in one hash-chained log — so you replay the whole hive to byte-identical state (zero model calls) and attribute any failure to the exact agent. Replay a hive. Find the culprit.

After any run, replay the whole swarm from the recorded decision log — into byte-identical state, with zero model calls — and attribute any failure to the exact agent and decision. "It only happens sometimes," for a hive of agents, becomes a normal debugging session.

Why it's different

Most frameworks let the model drive execution — a tool call is a side effect, so the one non-deterministic thing in your system is in the control path. Jaros inverts it: the model only proposes inert data; a deterministic executor decides what runs. Three properties fall out, and they're the whole point.

🔁

Reproducible by replay

Every decision is recorded before any effect. Replay reconstructs the run to byte-identical state with no model call — crash recovery is just a special case of replay.

🔒

Capability-safe by construction

Agents hold only the scoped handles you grant. A bad decision can't reach what it was never given, and every mediated action is written to a durable audit log.

📦

Zero infrastructure

No server, database, or broker — files and threads. Runs offline with no API key. A build check fails if any module so much as imports a database driver or broker.

Quickstart — under 5 minutes, no API key

It runs offline on a deterministic default adapter. Two terminals: one for the OS, one to drive it over the shared file system.

# 1. install — no API key, no database, no server
pip install jaros

# 2. boot the OS on a data dir (foreground — use a second terminal for step 3)
export JAROS_DATA_DIR=.jaros-data
jaros serve

# 3. second terminal: submit work, watch it run
export JAROS_DATA_DIR=.jaros-data
jaros submit greeter --input '{"name":"Jaros"}'
jaros watch

# 4. THE WOW: replay the whole run to byte-identical state, zero model calls
jaros replay

Honest about what it is

Over-claiming is epidemic in this space. Jaros claims only what the architecture delivers.

It is

  • A zero-infrastructure runtime — files and threads, runs anywhere files work.
  • A determinism-and-replay engine — reproduce the exact run from the decision log.
  • A capability-safe execution model — least-privilege handles, default-deny, audited.
  • The graduation layer between a prototype and cluster-scale infra.

It is not

  • Not a hardened security sandbox (host isolation stays the host's job).
  • Not a cluster-scale distributed system (single-node-first, bounded multi-node).
  • Not an agent-authorization / governance gateway.
  • Not "unbreakable" — it is durable, crash-recoverable, and replayable.

One honest caveat: byte-identical replay holds because executor handlers are deterministic — so Jaros makes that a checked invariant (a CI guardrail fails the build if the core path stops reproducing), not something you have to trust.
